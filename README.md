# 国会 法律案トラッカー（プロトタイプ）

提出された法律案のステータスを自動整理し、参考情報を付して見やすく提供するサイトのデザイン検証用プロトタイプ。

## 収録サンプル（第221回国会・特別会 / 令和8年）

| 議案 | 法案 | 所管 | ステータス（2026-06-17時点） |
|---|---|---|---|
| 閣法31 | 小型無人機(ドローン)飛行禁止法 改正案 | 警察庁 | **審議中**（参・本会議採決待ち） |
| 閣法33 | 携帯電話不正利用防止法(本人確認) 改正案 | 総務省 | **成立**（5/29公布・法律25号） |
| 閣法32 | 海外通信・放送・郵便事業支援機構(JICT)法 改正案 | 総務省 | **成立**（5/7公布・法律18号） |
| 衆法58 | 郵政民営化法 改正案（議員立法・第217回提出） | 議員提出 | **継続審査** |

成立2件・審議中1件・継続審査1件で、ステータスの多様性を意図的にカバー。

## 実装している要素

- **ダッシュボード**：ステータス別の件数集計
- **フィルタ**：ステータス／提出区分（閣法・衆法・参法）／所管省庁／キーワード（クライアント側で即時絞り込み）
- **法案カード**：ステータスバッジ・議案番号・所管・成立確度・タグ
- **詳細ドロワー**：
  - **概要（AI生成）** … 提案理由・趣旨をもとにした要約（「要確認」注記付き）
  - **審議タイムライン** … 提出→衆委→衆本→参委→参本→公布。現在地・可決・継続を色分け
  - **参考情報（信頼度別）** … Tier1 一次/準一次 → Tier2 立法補佐機関 → Tier3 シンクタンク → Tier4 専門家解説。自動紐付けの確信度バー付き

## ファイル

- `index.html` … UI・描画ロジック（依存ライブラリなし。`data_collected.js` を読み込む）
- `collect.py` … 収集スクリプト（参議院 議案情報をスクレイピング → 正規化JSON。会議録APIリンクも生成）
- `enrich.py` … 補強モジュール（内閣法制局・衆議院から 所管省庁・提出理由 を付与）
- `sources.yaml` / `crawl.py` … 参考リンクのソース台帳とクローラ（→ `refs.db`）
- `aliases.yaml` … 法案の通称・関連キーワード辞書（審議会等との紐付けの鍵）
- `match_refs.py` … 参考文書を法案に紐付け（語彙＋所管＋趣旨＋意味類似でスコア化）
- `llm_gate.py` … 弱い候補をLLMで関連性検証（任意・要APIキー）
- `linkcheck.py` … 参考リンクの死活チェック（＋Wayback提案）
- `run_all.bat` … 日次パイプライン（収集→クロール→照合→死活）一括実行
- `tag.py` … 検索用タグ付与（ステータス／所管／法令名／主題キーワード）
- `ingest_submission.py` / `merge_submissions.py` … 参考情報の提供（Issue）処理と掲載反映
- `requirements.txt` … 依存パッケージ
- `DEPLOY.md` / `.github/workflows/` … 公開方法・CI自動更新・情報提供Issue処理
- `bills.json` / `data_collected.js` … パイプラインの出力（`window.BILLS` に直接流用可）
- `refs.db` … 収集した参考文書のインデックス（SQLite）

## 収集パイプラインの使い方

```
python collect.py --diet 221                      # 全法律案（省庁・提出理由も補完）
python collect.py --diet 221 --only 31,32,33,13   # 議案番号で絞り込み
python collect.py --diet 221 --no-enrich          # 参議院データのみ（高速）
python collect.py --diet 221 --llm-summary        # 提出理由をLLMで3行要約（要 ANTHROPIC_API_KEY）
python collect.py --diet 222 --clb-id <id>        # 新会期は内閣法制局の一覧IDを指定
```

処理段階:
1. **収集**（`collect.py`）… 参議院 議案明細（衆参両院の経過を1ページに含む）を解析
2. **正規化** … 先議区分で院順を決め、提出→委員会→本会議→公布のタイムラインを合成。
   公布日/継続/否決の有無から ステータス（成立・審議中・継続審査・廃案）を判定
3. **補強**（`enrich.py`）… 概要・所管・参考リンクを付与
   - **閣法**: 内閣法制局の一覧から **主管省庁**、詳細ページから **提出理由**
   - **衆法・参法**（議員立法）: 衆議院の提出法律案本文から末尾の **「理由」** を抽出
   - 概要は `--llm-summary` で**ローカルLLM(Ollama)が自分の言葉に要約**（著作権配慮で原文非転載、
     `summaries.json` にキャッシュ）。モデルは `OLLAMA_MODEL`（既定 gemma4:12b）。フラグ無しは原文
   - 所管省庁ページ・内閣法制局詳細ページ・衆議院本文を参考リンクに追加

## サイトでの表示

`index.html` は収集パイプラインの出力 `data_collected.js` を読み込んで表示する（単一データ源）。

## 参考リンクの収集

設計は `DESIGN_refs.md`。収集（クロール）と紐付け（マッチング）を分離している。

```
python crawl.py            # sources.yaml を巡回し refs.db に文書を蓄積（845件）
python crawl.py --stats    # 蓄積状況
python match_refs.py       # refs.db の文書を bills.json に紐付け → data_collected.js
python match_refs.py --dry # 書き込まずレポートのみ
python match_refs.py --no-semantic  # 意味類似を無効化
python match_refs.py --llm-gate     # 弱い候補をLLMで関連性検証（要 ANTHROPIC_API_KEY）
run_all.bat                # 収集→クロール→照合→死活 を一括（Windowsタスクスケジューラ向け）
```

公開方法（GitHub Pages + Actions 等）は `DEPLOY.md` を参照。

## 参考情報の提供（ユーザー投稿）

各法案の詳細から「参考情報を提供する」→ 事前入力済みの GitHub Issue（`submit-ref.yml`）で
**対象法案＋URL**（NDLサーチ/CiNiiの書誌、記事URL等）を投稿。以後は自動：

1. Issue投稿/編集 → `ingest_submission.py` が **書誌メタ（og/citation/JSON-LD/title）を抽出**し、
   `match_refs` の語彙＋趣旨キーワードで **関連度を推定** → 結果をIssueにコメント＋`needs-review`。
   （NDLデジコレ等のJS描画ページはタイトル自動取得不可→コメントで手動補完）
2. メンテナが内容を確認し `approved` ラベル → `submissions.json` に取り込み、
   `merge_submissions.py` がサイトデータへ「提供情報」(Tier4)として反映、Issueをクローズ。
3. 日次ビルドは match→**merge_submissions**→tag の順で、再生成後も提供情報を維持。

本文は保存せずタイトル・URL・発行日のみ（著作権配慮）。NDLはpidで重複排除。
国会図書館は**2024〜2026年**を巡回（法案は提出の1〜2年前から論点化されるため広めに）。
審議会は所管省庁が一致する会議のみ照合対象とし、誤マッチを抑制（文科省403/経産省・内閣府は未対応）。

### マッチャ（`match_refs.py` / `aliases.yaml`）

4系統でスコア化し Tier別閾値で採用（DESIGN_refs.md §5）:
- **語彙一致**（立法補佐機関）… 法案名の部分一致＋文字3-gram。「立法と調査」は
  副題に法案名を含むため高精度。例: 防衛省設置法/太陽電池廃棄物/金融商品取引法 = 77〜92%。
- **趣旨キーワード**（部分改正法に有効）… `summary`(提出理由)から内容語を抽出し、
  **コーパス文書頻度(DF)で希少語だけ採用**（IDF的フィルタ）して照合。件名が汎用
  （「民法等の一部を改正…」）でも趣旨の主題語で拾える。例: 成年後見/暗号資産/資金決済/
  郵政事業/プラスチック/災害対策基本法 等。
- **所管×通称**（審議会・検討会）… 所管省庁が一致する会議のみ対象（precision確保）に、
  `aliases.yaml` の通称キーワード＋趣旨キーワードで紐付け。会議ページは h1＋WG名＋報告書名を
  内容語として収録するため、抽象的な会議名でも拾える。例: ドローン法案→警察庁「違法な
  ドローン飛行対策に関する検討会」、携帯本人確認法→総務省「不適正利用対策WG/利用者情報WG」、
  郵便法→総務省「郵便料金政策委員会」= 82%。
- **意味類似**（プラガブル層）… `multilingual-e5`（導入済・最良）→ Voyage → TF-IDF文字n-gram
  の順で自動選択。**現在は e5 が有効**（`pip install sentence-transformers`）。同義語
  （太陽光パネル↔太陽電池, ドローン↔小型無人機, 本人確認↔不適正利用）を橋渡しできる。
  ただし短いタイトル同士では値域が圧縮され精度がまちまちのため、**法案ごとtop-1・
  平均からのマージン要件付きで「要確認候補」として控えめに採用**（confは70%上限）。
  `--no-semantic` で無効化可。例: 下水道法案 → 「下水道法・道路法等の一部改正」（件名一致で
  漏れたものをe5が補足）。

> 既知の限界: TF-IDFは文字の重なりしか見ないため、真の同義
> （太陽光パネル↔太陽電池 / 本人確認↔ガバナンス / ドローン↔小型無人機）は橋渡しできない。
> 当面は `aliases.yaml` が同義を吸収し、本格対応は `sentence-transformers` 導入で
> `build_semantic()` が自動的に e5 へ切り替わる（コード変更不要）。

## 本番化に向けたデータパイプライン対応

サイトデータ `data_collected.js` は、以下の自動収集パイプラインで生成している。

```
[収集]  参議院 議案情報 / 衆議院 審査経過（HTMLスクレイピング）
        国会会議録検索 API（JSON）… 質疑・論点
        内閣法制局 / 所管省庁 … 提案理由・要綱・新旧対照
            │  日次バッチ（weekly_check と同方式）
[正規化] イベント列 → ステートマシンで現在ステータス・成立確度を算出
[付与]   所管省庁×時期×テーマで審議会・検討会を逆引き
        主要シンクタンク／法律事務所のRSS・更新ページをインデックス化
        → 法案要約のベクトルとセマンティック検索で候補抽出＋確信度スコア
        → Tier1は自動採用、Tier3-4は「候補」表示
[配信]   静的サイト生成 or API + フロント（本プロトタイプのUIをそのまま流用可）
```

### 確信度と表示の出し分け（信頼性の肝）

- Tier1（議案情報・会議録・所管省庁・国会図書館）＝法案単位で紐付くため**自動採用**
- Tier3-4（シンクタンク・法律事務所）＝**候補＋確信度**で提示し、本文転載はせずリンク＋短い要約のみ
- AI生成物には必ず「要確認」注記と一次資料リンクを併記

## 注意

- 「概要」「参考情報の確信度」はプロトタイプ用のAI付与例。
- 一次資料リンクは検証済みURL。シンクタンク・法律事務所の項目は、実在記事URLの捏造を避けるため**自動検索クエリURL**で代替表示している（本番では収集済みインデックスの実URLに置換）。
